Choose your evaluator based on the nature of the failure mode. Default to code-based — only escalate when the criterion is genuinely subjective.

Decision: Which evaluator?
Code-based DEFAULT
→ use when: Objective, deterministic, rule-based failures
→ avoid when: Nuanced subjective criteria
e.g. Exact match, JSON validation, regex, classifiers
LLM-as-a-Judge
→ use when: Subjective criteria: tone, helpfulness, coherence, safety
→ avoid when: Human labels haven't been used to validate TPR/TNR first
e.g. Relevance scoring, hallucination detection, style eval
Human evaluator
→ use when: High-stakes decisions, calibrating LLM-as-a-Judge rubric, edge cases
→ avoid when: Scale — human eval is slow and expensive
e.g. Rubric calibration, golden set creation, edge case triage
Calibrate the LLM-as-a-Judge before deploying at scale
An LLM-as-a-Judge is an AI product in its own right — evaluate it before putting it to work. It needs its own golden dataset, separate from your AI product's.
→Collect human annotations on the judge's decisions, not the AI output's quality
→Curate for edge cases, ambiguous outputs, and annotator disagreements
→Use it to measure TPR/TNR and iterate on the rubric before the judge goes live
TPR = TP / (TP + FN)
Does the judge catch real failures? Low TPR = misses real issues.
TNR = TN / (TN + FP)
Does the judge avoid false alarms? Low TNR = over-triggers on good outputs.
LLM-as-a-Judge Prompt Template
PROMPT STRUCTURE: Role → Context → Task → Rubric → Score
Role: You are an expert evaluator assessing whether the retrieved context is relevant to answer a user's question. Your task is to evaluate how directly and completely the context supports a response to the question. Context: You will be given: 1. A user's question 2. A retrieved context passage Task: First, review the user's question: {USER_QUESTION} Now review the retrieved context: {RETRIEVED_CONTEXT} Evaluate the context for RELEVANCE: - Does the context contain information that directly addresses the question? - Is the key information present, or only tangentially related content? - Could an accurate answer be grounded in this context alone? - Is there missing information that would require additional context? Then assign a relevance score between 0 and 1: - 0 = Context is completely unrelated to the question - 0.5 = Context is partially relevant but missing key information to answer fully - 1 = Context fully supports an accurate and complete answer Return your evaluation in <score>[0–1]</score>

Your golden dataset is the single source of truth for your eval. Build it carefully — it directly determines what your evaluator can and cannot detect.

Golden Dataset Rules
✓Define good and bad with domain experts
✓Start with 50–100 production samples, annotated by humans
✓Mix: 60% good, 20% bad, 20% edge cases
✓Slice into subsets: by cohort, source, topic
✓Synthetic is OK — but base on real failure cases, avoid generic/repetitive
✓If using few-shot prompting, do train/test split
✗Don't annotate after seeing the score — annotate blind
✗Don't use generic samples disconnected from your domain
✗Don't skip re-annotation when pipeline changes significantly

Spot failures, not perfection
Don't expect perfect scores. Eval exists to surface failure patterns — not achieve 100%. A well-calibrated 0.70 beats an inflated 0.95.

Common pitfalls and principles that apply across all evaluation contexts.

✓Establish eval before development
✗Wait until after development to think about eval
✓Build failure test cases if no production data
✗Use vague, generic criteria
✓Be specific: domain + failure mode + metric
✗Over-rely on off-the-shelf evals
✓Use stronger models to judge weaker ones
✗Ignore biases in judge models
✓Start with binary metrics, then refine
✗Skip human alignment validation for the LLM-as-a-Judge
✓Iterate on rubric clarity with few-shot examples
✗Expect perfect scores — eval spots failures
✓Mind empty outputs in results
✗Use one evaluator for multiple failure modes
